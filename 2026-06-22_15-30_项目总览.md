## 工作区声明

2026-06-21 起，**省科协科普视频** 升级为独立工作主目录。Agent 进入项目后第一个读取 `./AGENTS.md`，而非上层 `general_chat/` 的全局规则。本目录下的 AGENTS.md 定义了四个文件的强制阅读清单和"任何修改前必先算 blast radius"的硬约束。与提案工作流相关的参考文件（Handoff 文档、Matt Pocock Skills 解析）已迁入本目录。

## 项目愿景

目标是：给定一篇科普文字稿，通过 AI 全链路自动化，生成一部完整的、接近差评君/冲浪Plus 质量水准的科普视频。

人类主要负责两件事：提供科普选题的文字稿，以及最终成片的质量审核与微调。中间的分镜脚本、素材清单、素材获取、程序化渲染代码和成片导出，都尽量交给 AI 按流程完成。

## 全链路生产流程

整个管线从零到成片共六个阶段。核心设计原则：每一步都用文本或结构化数据做中间产物，每一步都能独立重跑。不是为了第一次就完美——是为了每次改完上游都能从头自动再生。

### 阶段零：从好视频里提取格式

新视频不知道分镜脚本应该长什么样。不能凭空设计——得从已有好视频里学。

拿差评君的一期成品视频 → 用 v4 Prompt 指挥 AI 把它逐秒拆成 `时间轴 | 底层原片事件 | 后期干预手段 | 音频` 四列表格 → 得到一份拆解文档（当前：[AI集体涨价拉片拆解文档](./标杆视频库/差评君/C_AI集体涨价_抖音/2026-06-17_16-00_AI集体涨价_拉片拆解文档.md)）。

这份拆解文档有两层价值。第一，它记录了差评君是怎么组织画面的——什么时候切镜头、什么时候叠化、什么时候用高亮框、什么时候靠实拍素材带动情绪。第二，这个四列表格本身就是未来分镜脚本的**格式模板**。不是参考——是模板。新分镜要模仿的是这份拆解文档的结构（四个列）、粒度（Sequence 2-5 秒）、视觉语法（持久窗口壳、底部信息栏、鼠标指针、证据截图、高亮强调层），而不是 v4 Prompt 里面的规则文本。

当前 Few-Shot 状态：C 有 v4 完整拉片，D 为 v2 产物。所有新增和返工拉片统一用 v4 Prompt。

### 阶段一：用学到的格式写新分镜

拿到新旁白稿后，把旁白稿 + 拆解文档的 Few-Shot 样本一起喂给 AI。告诉它：**模仿拆解文档的画面节奏、视觉语法、表格结构，为新的旁白稿设计逐秒分镜。**

为什么要这样做？因为拆解文档已经把好视频的叙事规律（每段 30-40 秒一层论据、每 3-5 秒一次画面变化、抽象论点配具象视觉锚点）固化成了一套可模仿的格式。直接让 AI 对着格式填新内容，比让 AI 自己从空白设计分镜要靠谱得多。

输入：旁白稿 + Few-Shot 拆解样本 → 输出：逐秒分镜脚本（物理字典描述，不指定具体实现库）→ 存放：`视频项目/.../03_分镜脚本/`

### 阶段二：素材清单提取

阶段一产出的分镜脚本是给人读的设计蓝图——画面描述、设计意图、旁白节奏都有。但下一步是让 AI 去执行，AI 不能从一句"画面叠加科技感背景图"直接跳到"去 Pexels 搜 technology background"。它需要一份结构化工单：这个素材是什么类型、编号多少、能不能自动获取。

所以阶段二本质上是一个**翻译步骤**——把"画面设计语言"翻译成"资产获取指令"。分镜里每一个图层引用了一个素材，全部提取出来：编号、分类、写获取指令。

编号 `ASSET_[镜头序号]_[图层序号]` 不是随手标的——它是分镜和渲染之间的合同。第一个数字是镜头序号（对应分镜的时间段），第二个是图层序号（对应那个镜头里的第几层画面）。渲染代码用这个编号找素材文件，两边靠编号对齐。没有编号体系：分镜说"第三段中间那个图表的配图"，渲染代码不知道对应哪个 PNG 文件。

分类这一步直接决定后面能自动化到什么程度。如果分镜写的是纯代码画的元素——高亮框、弹窗、文字卡片——直接标 `[AUTO]`，不产生获取任务。其余分类（`AI_BROWSER_CAPTURE`、`STOCK_LIBRARY`、`AI_GENERATED`、`HUMAN_CAPTURE`）各自走不同的获取路径。这一步拆得清楚，下一步才能四条路径并行跑。

输入：分镜脚本 → 输出：两层——结构化 JSON（给 AI 执行）+ Markdown 概览（给人快速扫描）

> **注意：** 当前素材分类体系和阶段三的获取路径是初步设计，正在 P002 中迭代讨论。执行前查阅 `_proposals/P002_素材标签到资产协议翻译层设计.md` 的最新结论。

### 阶段三：素材自动获取

这是整条管线自动化程度最高的环节，也是"全自动 pipeline"目标的核心战场。理想情况下：阶段二清单里的每一条 `[TODO]` 都在这一步被 AI 自动解决——不需要人类打开浏览器截图、不需要去图库翻页亲手挑。

设计原则是**分类分流、并行执行、失败降级**：

- `GENERATE_BY_CODE`（已标 `[AUTO]`）：跳过。高亮框、弹窗、进度条、文字卡片——渲染阶段直接在 React 组件里用 CSS/SVG 画，不产生素材文件。
- `AI_BROWSER_CAPTURE`：浏览器自动化。AI 用 Playwright 打开目标网址，模拟必要交互（登录态、滚动到特定位置、等待异步加载完成），截取对应页面状态并保存为 `ASSET_xx_xx.png`。为什么不用人类手动截图？因为后续分镜可能迭代——旁白改了、镜头重排了——每次都需要可重现的截图。自动化保证一致性。
- `STOCK_LIBRARY`：图库 API。AI 按清单里的搜索关键词调用 Pexels/Unsplash/Pixabay 的 API，自动筛选匹配图片。清单里给的是一组关键词而不是一个固定链接——第一组搜不到自动切第二组、第三组，目标是尽量在这一步就搜到可用素材。
- `AI_GENERATED`：AI 图片生成。按清单给的 Prompt 调用生成 API 产出图片。
- `HUMAN_CAPTURE`：仅当 AI 确实拿不到时才标注——比如需要登录内部系统查阅的数据界面、需要签字授权才能使用的资料。目标是最小化这个类别的条目数：每多一个 `HUMAN`，管线就多一个断点。生成占位图保证流程不阻断，最终需要人类补上。

每完成一条素材，状态从 `[TODO: xxx]` 改为 `[DONE]`。渲染阶段启动时扫描清单——如果仍有 `[TODO]` 条目且对应文件不存在，脚本直接阻断告警，不拿错误占位图糊过去。

这步的终点状态：所有 AI 能自动获取的全部到位，只剩少数 `HUMAN` 条目等待人工处理。这就是自动化的边界——AI 做到的做到极致，做不到的不假装能做到。

> **注意：** 同上，阶段三的获取路径细节随 P002 迭代。执行前查阅最新结论。

### 阶段四：程序化渲染

分镜脚本有了、素材文件齐了——现在把它们组装成一段可播放的视频。选程序化渲染（Remotion）而非手动剪辑（Premiere/剪映），是因为整条管线需要**可重跑**。

设想这个场景：阶段二发现了更好的素材来源，改了几个搜索关键词，重新跑了一遍阶段三。手动剪辑意味着你得从头再拖一遍时间轴，把新素材对到原来的位置。程序化渲染意味着重新 `pnpm render`——代码自动把新素材放到分镜指定的时间段里。改动的部分自动体现在成片中，不改的部分保持原样。

这也是为什么阶段一强制只写物理字典、不写具体实现库。"Scale 70%→100% + Spring Damping 15" 在渲染时直接被翻译成 Remotion 的 `spring()` 调用——今天用 Remotion，明天换 Motion，后天用 GSAP，分镜脚本不需要改一个字。分镜只管"画面要怎么动"（纯物理量），渲染 Agent 自己决定"用什么代码实现这个动作"。

`共享工具/库对比测试/conclusion.md` 告诉渲染 Agent 每种物理效果最适合用哪个库——GSAP 主选、PixiJS 适合放大镜效果、anime.js 适合 ClipPath 擦除。这是从实验数据来的，不是拍脑袋选的。CODE 类元素——高亮框、弹窗、图表——全在 React 组件里直接绘制，它们不依赖任何外部素材文件。素材文件的引用通过 `public/assets/` 的 symlink 指到 `04_素材清单/assets/`，`asset-manifest.json` 维护编号到文件名的映射。

输入：分镜脚本 + 素材文件 + 库对比测试结论 → 输出：TypeScript 工程代码

### 阶段五：导片与复盘

`pnpm render` 输出 MP4。但这不是终点——是第一轮迭代的开始。

成片效果不满意，**不能直接修阶段四的代码**。手动调了就破了可重跑的前提。正确做法是判断问题出在哪一环，回到那一环修源头：

- 画面节奏不对（某段太快或太慢、信息密度没跟上） → 调阶段一的 Prompt 或扩充 Few-Shot 样本
- 某段素材质量差不搭调 → 回阶段三换搜索关键词或手动替换
- 动效生硬不自然 → 在阶段二分镜中细化运动描述，重新跑阶段四
- Few-Shot 拆解本身画风偏了 → 回到阶段零重拆，或拉更多标杆视频补充样本

因为中间产物全部是文本和结构化数据（分镜是 Markdown 表格、清单是 JSON、渲染是 TypeScript），每一步都可以独立重跑。改上游 → 顺流重跑 → 看新成片 → 判断是否有效。这就是程序化管道设计的核心优势：**不是一次生成完美，而是每次都能从头迭代到满意。**

## 已知风险

最主要的风险还是偏差逐层累积：拉片阶段模型可能误判复杂图层运动；Few-Shot 阶段模型可能分不清“优秀视觉语法”和“拉片噪音”；渲染阶段再把偏差翻译成代码，最终画面就会越来越偏。

应对方式是继续坚持三件事：拉片时强制原片/后期解耦；分镜时只写标准物理字典；渲染前用素材清单和工程标准做人工检查。
