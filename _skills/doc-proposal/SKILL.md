## Phase 1：追问（Grilling）

**规则：一次只问一个问题，每次给出推荐答案。等用户回答后再问下一个。不要一次提出多个问题。**

按以下分支顺序逐一确认。能从文件中直接读出答案的分支，直接说明并跳过，不要问用户已知的事情。

### 必须覆盖的决策分支

**A. 变更起点**
- 这次要改什么？（对照依赖关系图定位到哪条链）
- 是纯文档内容调整，还是涉及文件路径/目录结构变化？

**B. Blast Radius 确认**
- 对照依赖关系图的相关链，逐个列出受影响文件，逐一问用户：这个文件需要改吗？改什么？
- 有没有跨链影响？（例如工程标准改了 → 触发链 2，又因双向性触发链 1）

**C. 单一来源检查**
- 这次变更会不会让同一个信息出现在两个文件里？
- 如果会，哪个文件是权威来源，另一个怎么处理（引用 or 删除）？

**D. 不动的范围**
- 哪些文件/章节不需要动？（明确排除，防止误改）

**E. 完成条件**
- 做完这次变更，你会怎么验证结果是对的？

走完所有分支后，用 1-2 段话总结当前理解，等用户确认后进入 Phase 2。

## Phase 2：PRD 综合

**不再追问。直接基于 Phase 1 的确认内容综合成结构化文档。**

使用以下模板，写到聊天里（如果变更较小）或写到 BACKLOG.md 的临时 Proposal 区块（如果变更较大）：

```
### Proposal：[变更标题]

**问题陈述**
当前文档体系中，[什么东西] 存在 [什么问题]。

**目标状态**
变更完成后，[描述目标状态，不是操作步骤]。

**使用场景**
- 作为 [角色]，我需要 [做什么]，现在 [怎样会更好]。

**变更决策清单**
- [文件 A]：[做什么，为什么]
- [文件 B]：[做什么，为什么]
- ...（对照依赖关系图的链，逐文件列出）

**不动的范围**
- [文件 X]：不涉及本次变更
- [文件 Y]：不涉及本次变更

**完成验证方式**
- [用户在 Phase 1 说的验证方式]
```

---

## Phase 3：任务拆解

把 PRD 的"变更决策清单"拆成可独立执行的 BACKLOG 任务。

规则：
- 每个任务独立可完成（不是"改这个文件的前半段"）
- 标注前置依赖（哪个任务必须先完成）
- 标注 `[P]` 表示可与其他任务并行
- 关联依赖关系图的链编号（如"触发链 1 + 链 2"）

直接追加到 `BACKLOG.md` 的"就绪"队列。追加格式：

```markdown
### T-NNN：[任务标题] [P 可选]

- 触发链：链 N（+ 链 M）
- 前置依赖：T-XXX（或"无"）
- 影响文件：[文件列表]
- 预期产出：[具体描述]
- 子任务：
  - [ ] NNN-1 [具体操作]
  - [ ] NNN-2 [具体操作]
```

## Phase 4：执行与验证

**按 [AGENTS.md 的 Blast Radius 行为](./AGENTS.md#blast-radius-行为硬约束) 执行。** 不在本 SKILL 里重复行为规则。

对照 [依赖关系图](./2026-06-21_10-55_文档依赖与影响关系图.md) 的链数据逐文件执行。

每个任务完成后更新 BACKLOG.md 的任务状态。
