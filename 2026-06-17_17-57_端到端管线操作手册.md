# 端到端管线操作手册

从文字稿到 30 秒视频的分步操作指南。每一步由人工触发 AI Agent 执行，检查中间产物后再进入下一步。

## 管线总览

```text
步骤 0         步骤 1          步骤 2          步骤 3          步骤 4          步骤 5
标杆视频拉片 → 分镜脚本生成 → 素材清单提取 → 素材获取 → Remotion代码生成 → 渲染
(Gemini)      (Gemini)       (AI Agent)    (AI Agent)   (AI Agent)       (pnpm)
    ↓             ↓              ↓             ↓             ↓              ↓
 Few-Shot      人工检查       人工检查      人工检查      人工检查        看片
```

当前范围：**豆包科普视频前 30 秒**（旁白稿前 4 段，约 00:00-00:30）。

## 步骤 0：标杆视频拉片生成 Few-Shot 样本

这一步是“few shot 分镜头脚本拆解”的上游。要先把差评君/冲浪Plus 的标杆视频拆成高质量拉片样本，步骤 1 才能拿这些样本学习视觉语法。

当前统一使用这个分析 Prompt：

`共享工具/提示词/2026-06-17_15-50_Gemini工业级原片后期解耦拉片分析prompt_v4.txt`

### v4 Prompt 的核心要求

| 要求 | 检查方式 |
| :--- | :--- |
| 先拆“底层原片事件流” | 每个 Sequence 开头必须用一两句话说明录屏/实拍底图里完整发生了什么 |
| 再拆“后期干预手段” | 时间轴表必须说明运镜、UI 叠加、转场、音效如何依附于原片 |
| 强制素材打标 | 必须使用 `[SCREEN_RECORD]`、`[FOOTAGE]`、`[CODE]`、`[STOCK]`、`[BROWSER]` |
| 区分运动语法 | 录屏/实拍内部用 `Playback`、`Scroll`、`Cursor Move`、`Typewriting`、`Click`，不要用 `Translate` 描述时间流逝 |
| 按 Sequence 输出 | 不要切成 1 秒以内的碎片，每段建议 2-5 秒，覆盖一个完整动作单元或语义单元 |

### 输入

| 文件 | 路径 |
| :--- | :--- |
| 标杆视频原片 | `标杆视频库/.../raw_video/` |
| v4 分析 Prompt | `共享工具/提示词/2026-06-17_15-50_Gemini工业级原片后期解耦拉片分析prompt_v4.txt` |
| 拉片脚本 | `共享工具/拆解脚本/2026-06-10_21-00_openrouter_segment_analyze.py` |

### 输出

| 产物 | 存放路径 |
| :--- | :--- |
| 分段拉片结果 | `标杆视频库/.../分段拉片结果/Sxx_*.md` |
| 合并后的完整拉片 | `标杆视频库/.../完整拉片分析_merged.md` |

### 人工检查点

- 每个 Sequence 是否先交代底层连续事件流？
- 表格是否包含“底层原片事件 / 后期干预手段 / 音频轨道”？
- 是否把录屏内部动作写成 `Scroll`、`Typewriting`、`Click` 等，而不是误写成容器 `TranslateY`？
- 是否保留了后期干预为什么要这样做的解释？

## 步骤 1：分镜脚本生成

**执行者**：Gemini

**任务**：基于旁白稿 + Few-Shot 拉片样本，为豆包科普视频生成完整的逐秒分镜脚本。

### 输入

| 文件 | 路径 | 说明 |
| :--- | :--- | :--- |
| 旁白稿 | `视频项目/01_为什么他的豆包更聪明/00_文稿/2026-06-12_14-00_豆包科普视频_旁白稿_v0.2.md` | 全文约 50 行，本次只取前 4 段 |
| Few-Shot 样本 1 | `标杆视频库/差评君/C_AI集体涨价_抖音/完整拉片分析_merged.md` | 现有质量最高样本，基于 v2 Prompt 生成，仍可作为临时样本 |
| Few-Shot 样本 2 | `标杆视频库/差评君/D_信用卡APP_抖音/完整拉片分析_merged.md` | 第二份全长拉片样本，基于 v2 Prompt 生成，仍可作为临时样本 |
| 物理字典标准 | [工程标准：物理字典与素材协议](./2026-06-17_17-57_工程标准_物理字典与素材协议.md) | AI 必须遵守的运动描述规范 |

### Prompt 要点

这是一个**新 Prompt**，不同于步骤 0 的拉片 Prompt。步骤 0 是“看视频 → 分析画面”，步骤 1 是“读文字稿 + 学 Few-Shot → 设计画面”。

核心指令：

1. 角色：你是一个顶级的科普视频视觉导演，正在为一篇科普文字稿设计逐秒分镜脚本。
2. 学习 Few-Shot 样本的视觉语法，不是只学表格格式，而是学持久窗口壳、底部信息栏、鼠标指针、证据截图、高亮强调层、弹性物理动效这些画面组织方式。
3. 输出格式：先“持久画面架构”，再“时间轴分镜表”。
4. 强制约束：只使用物理字典描述动画，不指定任何实现库；每个素材必须标注分类标签；每段运动必须附带设计意图；每 3-5 秒至少一次可见 UI 状态变化。
5. 内容方向：豆包科普视频的视觉主题是“AI 产品 vs 模型”的认知科普，应大量使用豆包 App 界面截图、对比卡片等具象证据。

### 输出

| 产物 | 存放路径 |
| :--- | :--- |
| 分镜脚本 | `视频项目/01_为什么他的豆包更聪明/03_分镜脚本/2026-MM-DD_HH-MM_豆包科普_0-30s_分镜脚本.md` |

### 人工检查点

- 分镜是否覆盖旁白稿前 4 段全部文字？
- 每 3-5 秒是否都有可见画面变化？
- 是否有“持久画面架构”？
- 素材标签是否齐全？
- 运动描述是否只使用物理字典，没有出现库私有 API？
- 设计意图是否有实质内容？

## 步骤 2：素材清单提取

**执行者**：AI Agent

**任务**：从步骤 1 的分镜脚本中提取所有引用素材，逐个归类、编号、写获取指令。

### 输入

| 文件 | 路径 |
| :--- | :--- |
| 分镜脚本 | 步骤 1 的产出文件 |
| 素材 To-Do 协议 | [工程标准：物理字典与素材协议](./2026-06-17_17-57_工程标准_物理字典与素材协议.md) 中“统一素材 To-Do 协议”章节 |

### AI Agent 指令

```text
请阅读这份分镜脚本，按照“统一素材 To-Do 协议”的规范，提取所有素材并生成结构化清单。

对每个素材：
1. 分配编号：ASSET_[镜头序号]_[图层序号]
2. 标注分类：GENERATE_BY_CODE / AI_BROWSER_CAPTURE / STOCK_LIBRARY / AI_GENERATED / HUMAN_CAPTURE
3. 写获取指令：
   - [CODE] 类：标注 [AUTO]，无需外部获取
   - [BROWSER] 类：说明需要截取哪个网页的什么状态
   - [STOCK] 类：给出 3 组搜索关键词
   - [AI_GEN] 类：给出生成 Prompt、风格、构图要求
   - [HUMAN] 类：说明为什么 AI 无法获取
4. 标注当前状态：[AUTO] 或 [TODO: xxx]

输出为 JSON 文件，同时生成一份人类可读的 Markdown 概览。
```

### 输出

| 产物 | 存放路径 |
| :--- | :--- |
| 素材清单 JSON | `视频项目/.../04_素材清单/2026-MM-DD_HH-MM_豆包科普_0-30s_素材清单.json` |
| 素材清单概览 | `视频项目/.../04_素材清单/2026-MM-DD_HH-MM_豆包科普_0-30s_素材清单.md` |

## 步骤 3：素材获取

**执行者**：AI Agent

**任务**：按素材清单逐个获取素材文件。

### 输入

| 文件 | 路径 |
| :--- | :--- |
| 素材清单 JSON | 步骤 2 的产出文件 |
| 推荐图库工具 | [工程标准：物理字典与素材协议](./2026-06-17_17-57_工程标准_物理字典与素材协议.md) 中“推荐图库工具”章节 |

### 素材类型处理

`[CODE]` 类跳过，步骤 4 用代码直接绘制。

`[BROWSER]` 类使用 Playwright 或其他浏览器自动化工具访问目标网页、调整页面状态、截图保存为 `ASSET_xx_xx.png`。

`[STOCK]` 类使用素材清单中的搜索关键词调用 Pexels、Unsplash、Pixabay 或 Openverse，筛选后保存到 `04_素材清单/assets/`。

`[AI_GEN]` 类使用素材清单中的生成 Prompt 调用图片生成 API。

`[HUMAN]` 类标记为待人工处理，必要时生成占位图，确保渲染流程不阻断。

### 输出

| 产物 | 存放路径 |
| :--- | :--- |
| 素材文件 | `视频项目/.../04_素材清单/assets/ASSET_xx_xx.png` |
| 素材台账 | `视频项目/.../04_素材清单/assets/asset_ledger.json` |

## 步骤 4：Remotion 代码生成

**执行者**：AI Agent

**任务**：基于分镜脚本和已获取素材，生成可运行的 Remotion 工程代码。

### 输入

| 文件 | 路径 |
| :--- | :--- |
| 分镜脚本 | 步骤 1 的产出文件 |
| 素材文件 | `视频项目/.../04_素材清单/assets/` |
| 现有 Remotion 工程 | `视频项目/.../05_渲染工程/2026-06-03_10-20_remotion_豆包科普视频_0_30s工程样片/` |
| 库映射表（可选） | `共享工具/库对比测试/conclusion.md` |

### AI Agent 指令

```text
请基于以下分镜脚本，在现有 Remotion 工程中生成可运行的视频组件代码。

要求：
1. 参考现有工程结构和组件命名
2. 分镜脚本中的物理字典参数直接翻译为 Remotion interpolate/spring 调用
3. 素材文件通过 public/assets/ symlink 引用
4. [CODE] 类素材在 React 组件中直接用 CSS/SVG 绘制
5. 确保 TypeScript 类型正确，无编译错误
6. 更新 Root.tsx 注册新的 Composition
```

## 迭代循环

完成一轮后，根据步骤 5 的观感评价回到上游调整：

- 分镜质量不行 → 调整步骤 1 的 Prompt 或 Few-Shot 样本
- 素材质量不行 → 调整步骤 3 的搜索策略或手动替换素材
- 动效质量不行 → 在步骤 4 中细化指令或手动调整代码
- Few-Shot 样本本身有问题 → 回到步骤 0，用 v4 Prompt 重新拆解或补充更多标杆视频

这就是“改上游 → 看结果 → 判断是否有效”的快速反馈回路。
