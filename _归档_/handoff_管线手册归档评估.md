# Handoff: 管线手册归档 + 物理字典废除评估

> 创建时间：2026-06-23 11:00（北京时间）
> 来源：P001 讨论过程中用户提出的关联诉求

## 这是什么

P001（工程标准可读性重构）讨论时，用户提了一个更远的问题：端到端管线操作手册可能已经过时了，考虑归档——如果管线手册归档了，那工程标准里的物理字典定义是不是也可以废除了？

这个问题太大，不能塞进 P001——它触发的依赖链（2/3/4/5/6）会波及 8-12 个文件，而且核心决策（"管线手册到底过时了没有"）需要人类自己判断。所以把这个问题从 P001 剥离，写成这份 Handoff，让你（或下一个 Agent）在 P001 执行完毕后重新打开评估。

## 当前管线手册到底在管什么

`2026-06-22_15-30_端到端管线操作手册.md` 定义了从文字稿到成片的 6 步操作指南：

```
步骤 0：标杆视频拉片 → 步骤 1：分镜脚本生成 → 步骤 2：素材清单提取
→ 步骤 3：素材获取 → 步骤 4：Remotion 代码生成 → 步骤 5：渲染
```

每一步写了三个东西：输入路径（具体文件在哪）、AI Agent 指令（跟 AI 说什么让它做这步）、人工检查点（人看完什么才算这步通过）。

但它不是唯一的管线信息来源——项目总览里有一个流程图描述了同样的事情，BACKLOG.md 里有当前冲刺的具体任务。管线手册多做的事情是：给出了每一步的精确文件路径和给 AI 的口令文本。

## 它可能过时在哪

从对话上下文推测，可能的问题：

- **文件路径硬编码**：管线手册里写的路径是豆包科普视频 v0.2 的旁白稿路径，换一个视频项目就得全改。把它当成通用操作手册用，路径就成了误导
- **步骤 0（拉片）现在主要由项目总览和 Backlog 驱动**：拉片这件事已经不需要管线手册告诉 Agent 怎么做——Agent 读项目总览就知道 v4 Prompt 在哪、样本在哪
- **步骤 4-5（代码生成+渲染）还没跑通过**：手册里的 Remotion 指令是"假设会这样"，实际管线还没完整验证过，手册里的指令可能是错的
- **与 BACKLOG.md 职责重叠**：BACKLOG 已经有 T-001 子任务拆解了分镜/素材/渲染的具体操作，管线手册的信息可能已经被 Backlog 覆盖

但也要反过来想：
- 步骤 1-3 里的 AI Agent 指令是从实践调出来的（给 AI 的口令文本），删除后需要从 Git 历史里恢复
- 物理字典约束目前只在工程标准和管线手册里被引用——如果管线手册归档了，物理字典的唯一消费者就是"AI 在生成分镜时被 Prompt 要求遵守"，这个约束可能变弱

## 如果确认归档，会发生什么

**直接影响：**
- `端到端管线操作手册.md` 移出核心文档圈（从 AGENTS.md 强制阅读清单移除）
- 依赖关系图的链 2/3/4/5/6 中所有指向管线手册的引用需要清理或重定向

**连锁影响：**
- 物理字典的定义（工程标准里的空间定位、变换字典、流媒体语法等章节）失去了管线手册这个主要消费方。如果 AI 生成分镜的 Prompt 也不引用物理字典了，这些定义就变成纯参考，没有强制约束力——这就是用户说的"物理字典可能废除"的含义
- 素材 To-Do 协议（工程标准里 ASSET_xx_xx 编号规则、TODO/DONE 状态标记）同样失去管线手册这个执行方。Agent 拿到分镜脚本后，按什么规则提取素材就没了明确指令
- 推荐图库工具列表（Pexels/Unsplash 优先级）仍然可以作为 Agent 的常识参考存在，不依赖管线手册

**不在影响范围内：**
- 拉片素材标签和场景转场术语（这两个是 v4 Prompt 的镜像定义，不依赖管线手册）
- 已有分镜脚本和素材清单（它们是产物，不是规则）

## 建议的推进方式

在 P001 执行完毕、工程标准文件样式稳定之后，单独提一个 `/proposal`。到时候 Agent 会重新读所有核心文件，基于最新状态做判断。你可以先想这两个问题：

1. 管线手册里的哪类信息你现在还在用？哪类已经不需要了？
2. 物理字典是"肯定要保留"还是"可以考虑简化"？

这份 Handoff 到时候直接喂给 Agent 做 context。
