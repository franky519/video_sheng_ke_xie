## 这个选题适合怎么做

这条视频最好不要做成“技术新闻播报”，也不要做成“豆包测评”。它更适合做成一个普通用户能立刻理解的数字素养科普：同样是豆包，为什么你和别人得到的答案差很多？普通用户看完之后能带走一个动作，就是以后做复杂任务前先检查模式，不要默认用快速模式。

适合的表达方式是“差评君式日常问题钩子 + 徐冲浪/冲浪普拉斯式底层逻辑拆解”。开头用一个人人会遇到的具体场景，不上来讲模型名；中段再拆“产品”和“模型”的区别；最后落回使用建议。这样既有趣，也符合省科协短视频对科学性、知识性、通俗性、艺术性、趣味性的要求。

要特别注意措辞边界。火山引擎官方在豆包大模型 2.0 发布文章中明确说，豆包 2.0 系列包含 Pro、Lite、Mini 和 Code，并且个人用户可以通过豆包 App 的“专家”模式体验新模型，但它没有公开说“快速模式一定对应哪个具体底层模型”。所以视频里可以讲“快速模式和专家模式会把你带到不同能力档位”，但不宜死说“快速就是某某模型”。可以用“根据官方模型分层和实际体验推测”来处理。

## 风格参考

我没有直接下载 B 站完整视频。这个不建议做，一是版权和平台规则都不稳，二是我们现在只需要学习叙事和画面方法，不需要拿原片素材。CDP 浏览器这次也没有连上，所以这里只基于公开页面、公开报道和可访问信息做风格归纳。后续如果你给 3 到 5 个具体链接，我可以在不下载整片的前提下，用公开视频页面、截图和可访问片段继续细拆。

差评君比较适合学它的开场方式：从一个很日常、甚至有点荒诞的问题切进去，比如“为什么充电谣言每年都要火一次？”或者“为什么大家都想上暗网？”这类标题都不是“讲解某技术原理”，而是先抓住用户已经有的困惑，再把它拆开。画面上一般会混合真实网页、产品截图、表情包、夸张字幕和快节奏剪辑。我们这个选题可以对应成：“为什么同一个豆包，他的答案比你的聪明？”

徐冲浪/冲浪普拉斯这一类更适合学它的“知识降维”。公开报道里对这类 B 站内容视频的总结很准确：把难懂的商业、科技、资本逻辑融入吐槽、八卦和具体人物故事，用最简洁的语言讲清前因后果。这个方法可以用在“模型路由”和“推理成本”上。不要直接说“产品根据路由策略选择模型”，而要说成“同一家店，有快餐窗口，也有主厨窗口，你走错窗口，吃到的东西当然不一样”。

## 内容农场流水线可以借鉴什么

Jike 摘录里最有用的一条是 2026-02-03 的“知识区视频内容制作流水线”：先让 Claude 生成初稿，再对话生成“文稿对应配图”的 MD 文件，让 Manus 或字节 anygen 找对应图片，最后把配音和配图导入剪映完成视频。这个思路很适合我们，但不要照搬成“全自动成片”。我们需要的是“自动化打底，人工把关科学性和节奏”。

外部开源项目也能说明现在内容农场常见管线是什么。`youtube-shorts-pipeline / Verticals v3` 的 README 里把流程拆成 research、script、visuals、voice、captions、assemble、upload，并用 Edge TTS、Whisper 和 ffmpeg 串起来。`MoneyPrinterTurbo` 也是同类：输入主题或关键词，自动生成文案、素材、字幕、背景音乐并合成短视频。这些项目能证明一件事：技术上已经不难，真正难的是选题、事实核查、画面品味和版权边界。

## 增量调研：VidRush 和海外内容农场

这次沿着 VidRush 往下查，结论比“找一个 AI 视频生成器”更有用。VidRush 的本质不是让视频模型凭空生成全部画面，而是把海外 faceless YouTube 的团队制流程产品化：你给选题和 brief，它做脚本、旁白、素材匹配、字幕、缩略图和粗剪；生成后再用内置编辑器替换素材、调音频、改字幕、加文字和转场。它官方文档里也写得很清楚，适合 narration-over-B-roll 这种“旁白讲故事，画面辅助说明”的视频，不适合录屏教程、复杂 3D 动画、真人出镜 vlog、reaction/compilation，也暂时不支持非英语内容。这个边界对我们很关键：省科协这个视频要借鉴它的生产线，而不是依赖它直接成片。

VidRush 真正值得学的有四件事。第一是“先验证题材能不能被画面支撑”，它叫 B-roll availability test。换成我们的话，就是每一句旁白都要能找到画面：截图、UI 模拟、示意图、数据图、隐喻图，至少要有一种能落地。如果一句话只能放大段文字，这句就不适合进视频。第二是“参考视频只学文风，不学画面”。VidRush 的 Reference Video 会分析 YouTube 视频 transcript，学习节奏、句长、词汇和叙事语气，但它不控制 B-roll、转场和视觉风格。所以我们用差评君、徐冲浪做参考时，也应该只拆它们的开场钩子、吐槽密度、解释节奏和转折方式，不能搬画面素材。第三是“Auto 测试，Manual 精修”。它官方明确区分 Auto Mode 和 Manual Mode：Auto 适合快速批量测试，Manual 适合高控制、高准确度、带 reference video 的成片。我们的做法也应该是先让 AI 批量出 3 个开头、3 个标题、3 套视觉母题，再人工定稿。第四是“把 AI 当制作团队，而不是神谕”。VidRush 的提示词指南强调 director brief、style guide、target audience、key facts 四块，这和我们做科普视频完全一致：先讲清楚给谁看、想让他看完做什么、哪些事实必须准确、风格像什么。

Noah Morris / NexLev 这条线也能解释为什么 VidRush 会长成这样。Noah Morris 自述来自 faceless YouTube 和内容农场行业，后来做 NexLev、VidRush 这类后端工具。NexLev 官网强调的是找 niche、看频道数据、Chrome Extension、自动化工作流和多频道管理。也就是说，海外内容农场的核心不是“我有一个最强生成模型”，而是一个增长流水线：找最近爆的视频，拆标题和缩略图，拿 transcript 分析结构，找相似但未被充分覆盖的新题材，然后把脚本、旁白、素材、缩略图交给团队或工具快速生产。VidRush 官方 playbook 甚至直接讲 80/20：多数视频只是打平或小赚，靠少数爆款覆盖成本，所以他们追求的是速度和量。

但这套东西不能原样搬到省科协视频。内容农场为了 YouTube 收益，可以接受大量“good enough”的片子；我们现在是要交一个 3 到 5 分钟的科普作品，评审会看科学性、表达清晰度、审美和版权。YouTube 官方 2025 年 7 月也把 repetitious content 更新为 inauthentic content，明确把批量生产、模板化、缺少差异的内容列为不适合变现的方向。Reddit 上对 VidRush 的反馈也不是单向吹捧，有人说它能提高产量，也有人抱怨价格、积分、音频问题和账户处理。Trustpilot 的评价量也很小，而且是 claimed profile，所以只能作为弱参考。对我们来说，最稳的判断是：VidRush 路线说明“流程自动化”成立，但“完全自动成片”不是我们该追求的质量标准。

把它翻译成我们这个豆包科普视频，可以变成下面这条生产线：

| VidRush/内容农场环节 | 它原本解决什么 | 我们怎么借鉴 | 推荐工具 |
|---|---|---|---|
| Niche / Outlier Research | 找近期爆款和可复制结构 | 找差评君、徐冲浪一类参考视频，只拆钩子、节奏和解释方式 | B站页面、手动看片、Markdown 摘要 |
| B-roll Availability Test | 判断一个题材有没有足够素材 | 每句旁白都配一个画面方案，缺画面的句子删掉或改写 | `shots.csv`、素材清单 |
| Transcript / Reference Style | 从成功视频 transcript 学文风 | 把参考风格总结成“开场方式、句长、吐槽密度、转折节奏” | LLM、人工审阅 |
| Director Brief | 给 AI 明确创作目标 | “给普通用户解释为什么同一豆包回答差很多，看完会检查模式” | Markdown brief |
| Style Guide | 控制语气 | 差评君式问题钩子 + 徐冲浪式底层逻辑拆解，但少一点攻击性 | 旁白稿 |
| Key Facts | 降低幻觉 | 官方资料、源文稿、火山引擎文档人工核查 | Markdown 引用表 |
| Asset Manager | 找视频、图、B-roll | 优先 UI 模拟、截图、示意图、授权素材、AI 隐喻图 | Playwright、Remotion、图片生成、可商用素材库 |
| Voice / Narration | 自动配音 | 样片可用 TTS，正式片优先真人，因为这种风格靠语气 | 真人录音、edge-tts、剪映、ElevenLabs |
| Editor | 快速粗剪和替换 | 用 `shots.csv` 控制时间线，先代码化样片，再决定是否进剪映精修 | Remotion、FFmpeg、MoviePy、剪映 |
| Thumbnail / Packaging | 提高点击率 | 比赛视频也要有封面，但不要标题党；主视觉可以是“两种豆包回答差异” | Canva/PS/本地设计 |

更具体一点，我们可以把“内容农场”里最有价值的部分拆成一个可执行工作包：先做一份 `素材需求.md`，每个镜头写清楚“画面是什么、素材从哪来、是否需要生成、是否有版权风险、运动方式是什么”。这比直接让 AI 生成视频更可靠。比如“模型路由”这句，不要找一段泛泛的服务器视频，而是做一个豆包 App 图标背后分出 Pro、Lite、Mini、Code 四个引擎的动画；“成本分流”这句，不要找随机机房，而是做一个 GPU 账单和快慢通道的可视化。这样画面是为解释服务的，不是为了填满时间线。

这条线也提醒我们，下一步不应该先碰复杂工程，而是先把脚本变成素材任务。只要 `旁白稿 + shots.csv + 素材需求.md` 三件套稳定，后面无论用 Remotion、MoviePy、剪映，还是临时接一个 VidRush/同类工具，都不会失控。

## 推荐技术路线

这个视频不需要完全依赖 AI 视频模型。反而更适合用“可控画面”做出来，因为主题是产品模式、模型分层和用户操作，不是需要真实电影感的叙事。

最稳路线是：

| 环节 | 推荐工具 | 为什么 |
|---|---|---|
| 脚本和分镜 | Markdown / CSV | 方便逐镜头控制时间、旁白、画面和素材 |
| UI 画面 | 本地 HTML + Playwright 截图，或者真实产品截图 | 能做“同一问题，两种回答”的可控对比 |
| 动态图形 | Remotion | 用 React 写视频，适合做卡片、弹幕、缩放、时间轴和模型分层动画 |
| 解释性动画 | Manim | 适合做“产品层、模型层、路由箭头、成本曲线”这类干净动画 |
| 简单拼接 | MoviePy | 如果只做图片、字幕、音频拼接，比 Remotion 更轻 |
| 最终编码 | FFmpeg | 稳定控制 1080p、码率、文件大小和字幕烧录 |
| 配音 | 真人优先，TTS 备选 | 差评君和徐冲浪的风格都很依赖“人味”，真人更好 |
| TTS 备选 | edge-tts / 剪映配音 / 付费 TTS | edge-tts 可直接产出 mp3 和 srt，成本低 |
| 字幕 | 手写时间轴 / Whisper / edge-tts SRT | 已有脚本时可以直接生成字幕，再微调 |

| [Whisper](https://github.com/openai/whisper) | 语音识别，可用于给真人配音自动生成字幕草稿 |

## 画面风格定调

画面不要做成大段 PPT。比较合适的基调是“屏幕录制 + 对比卡片 + 手绘式箭头 + 快速吐槽字幕”。色彩上可以用黑白灰作为底色，强调色用豆包红、警告黄和信息蓝。字幕不要太花，重点词可以用黄色或红色高亮。每 5 到 8 秒必须有一次画面变化，哪怕只是推近、左右切换、弹出一个标签。

这个选题的核心画面母题有四个：

| 母题 | 画面做法 |
|---|---|
| 两个人问同一个 AI | 左右两部手机或两个聊天窗口，同一个问题，不同回答质量 |
| 豆包是产品，不是模型 | 一个 App 图标背后分出多个引擎/厨房窗口 |
| 快速和专家是两个入口 | 左边“快速通道”人多、便宜、快；右边“专家通道”更慢但更细 |
| 成本分流 | GPU 账单、硬币、排队通道、服务器电表这些视觉隐喻 |

## 旁白压缩方向

源文稿可以保留四个核心段落，其它内容改成画面信息。

第一段保留“同样问豆包，答案差很远”的钩子。

第二段保留“豆包是产品，不是模型”的解释。

第三段保留“快速和专家背后是不同能力档位”的判断，但要加一句“具体底层模型以官方公开信息为准，不做死断言”。

第四段保留“为什么默认不给最强：成本分流”和“复杂任务前先切专家”的行动建议。

不建议保留太多榜单排名。榜单变化太快，且 2026 年 5 月的 Arena 排名已经和文章里“前十、35 位”等说法不完全稳定。视频里可以说“榜单会快速变化”，但不要把具体排名当成核心论据。核心论据应该是官方模型分层、成本差异和用户可操作的模式选择。

## 可执行制作流程

第一步，把源文稿改成 1500 字左右的旁白稿。旁白稿每一句都要能对应一个画面，不要出现连续 20 秒只能放文字的段落。

第二步，把上面的镜头脚本落成 `shots.csv` 或 `shots.json`。字段至少包括：`start`、`end`、`voiceover`、`visual_type`、`asset`、`motion`、`notes`。

第三步，先做本地模拟 UI，不急着抓真实豆包界面。比如用 HTML 做两个聊天窗口，一个“快速模式”，一个“专家模式”，这样画面可控，也不会因为产品界面变化导致重做。

第四步，用 Remotion 做主工程。所有镜头都按照时间轴写成组件，卡片、箭头、缩放、字幕、BGM 都在代码里控制。需要解释“路由”和“成本分流”的地方，可以单独用 Manim 输出透明背景或黑底动画片段，再导入 Remotion。

第五步，配音。优先真人配音，因为这个风格靠语气。TTS 只适合第一版样片。用 edge-tts 的话，可以先生成 mp3 和 srt，再在 Remotion 或 ffmpeg 里合成。

第六步，用 ffmpeg 导出 1920x1080、16:9、MP4。省科协要求 1080p 以上、100 到 300 MB，这个可以靠码率控制。4 分钟视频如果用 H.264，目标码率大概 4 到 8 Mbps 就够。

第七步，做三轮 QA。第一轮查事实，特别是官方没有说过的话不要说死。第二轮查版权，B 站参考视频不能直接搬画面，素材要用自制图、授权图、公开可商用素材或自己录制的界面。第三轮查观感，每 5 到 8 秒有没有画面变化，字幕有没有挡住重点，普通人能不能看懂。

## 下一步建议

最值得先做的是两份文件：

`2026-05-25_豆包科普视频_旁白稿_v0.1.md`

`2026-05-25_豆包科普视频_shots.csv`

旁白稿定下来后，再决定是走 Remotion 工程，还是先用剪映/MoviePy 做低成本样片。我的建议是先做 Remotion 或 HTML 样片，因为这个题有大量 UI 和图形对比，代码生成画面比手动剪辑更稳，后续要改字、改节奏、改镜头也更快。
