## 项目愿景

目标是：给定一篇科普文字稿，通过 AI 全链路自动化，生成一部完整的、接近差评君/冲浪Plus 质量水准的科普视频。

人类主要负责两件事：提供科普选题的文字稿，以及最终成片的质量审核与微调。中间的分镜脚本、素材清单、素材获取、程序化渲染代码和成片导出，都尽量交给 AI 按流程完成。

## 全链路生产流程

```text
文字稿（人类提供）
    │
    ▼
第一步：分镜头脚本生成
输入：文字稿 + Few-Shot 拆解样本
输出：逐秒分镜脚本（物理字典描述，不指定具体实现库）
存放：视频项目/.../03_分镜脚本/
    │
    ▼
第二步：素材清单提取
输入：分镜头脚本
输出：结构化素材待办清单（编号/分类/获取指令）
存放：视频项目/.../04_素材清单/
    │
    ▼
第三步：素材自动获取
界面截图 → 浏览器自动化访问并截取
通用素材 → 图库 API 检索 + AI 筛选
代码素材 → 渲染代码直接生成
最终素材：视频项目/.../04_素材清单/assets/
    │
    ▼
第四步：程序化渲染合成
输入：分镜脚本 + 素材 + 库对比测试结论
输出：完整视频（Remotion 渲染）
成片：视频项目/.../06_成片/
```

数据流可以简单理解为：

```text

  └─ 共享工具/提示词（拉片 Prompt）

        ↓

        ↓

        ↓

        ↓

```

## 第一步：分镜头脚本生成

分镜头脚本是整条链路的核心产物。它不是粗略写“这里弹出一个卡片”，而是要把画面拆成可生产的图层、素材、运动参数、配音对齐和叙事意图。

分镜脚本只使用物理字典描述动画，例如 `Scale`、`Opacity`、`TranslateX/Y`、`Rotate`、`Blur`、`ClipPath`、`Spring`、`Bezier`。具体用 Remotion、GSAP、CSS 还是其他库，是第四步渲染阶段的事情。

### 输入

| 输入 | 来源 | 说明 |
| :--- | :--- | :--- |
| 文字稿 | `视频项目/.../00_文稿/` | 人类提供的科普选题文案 |
| 标杆视频 Few-Shot 拆解样本 | `标杆视频库/` | 差评君/冲浪Plus 等标杆视频经多模态拉片后的结构化脚本，AI 从中学习视觉语法 |
| 工程标准 | [工程标准：物理字典与素材协议](./2026-06-17_15-50_工程标准_物理字典与素材协议.md) | 约束运动描述和素材标签，防止 AI 生成不可实现的私有 API 或混乱素材类型 |

### Few-Shot 上游：标杆视频拉片

Few-Shot 样本不是手写出来的，它的上游是“标杆视频拉片分析”。这个环节现在统一使用新版 v4 Prompt：

`共享工具/提示词/2026-06-17_15-50_Gemini工业级原片后期解耦拉片分析prompt_v4.txt`

v4 的核心变化是把视频拆成两层看：先看真实底层原片里发生了什么，再看后期如何用运镜、UI 动效、转场和音效“寄生”在原片上。它要求按 Sequence 划分逻辑段落，不再把整段视频切成 1 秒以内的机械碎片。

当前拉片要求重点包括：

| 要求 | 实际含义 |
| :--- | :--- |
| 素材打标 | 强制区分 `[SCREEN_RECORD]`、`[FOOTAGE]`、`[CODE]`、`[STOCK]`、`[BROWSER]` |
| 原片/后期解耦 | 先描述底层连续事件流，再描述后期叠加如何辅助、掩盖或强调原片 |
| 运动语法分流 | 录屏和实拍视频内部只能用 `Playback`、`Scroll`、`Cursor Move`、`Typewriting`、`Click` 等流媒体/微交互语法，不用 `Translate` 误写时间流逝 |
| 场景级拆解 | 以 2-5 秒动作为基本粒度，按 Sequence 输出，不做 1 秒碎片化列表 |
| 结合表 | 每个场景必须输出“底层原片事件 / 后期干预手段 / 音频轨道”的结合分析 |

已有的 C、D 两个标杆拉片样本是在 v2 Prompt 下生成的，仍可暂时作为 Few-Shot 样本使用，因为它们格式完整、质量较高。但所有新增拉片和返工拉片，都应改用 v4 Prompt。

每个标杆视频最好做完整拆解，才能覆盖开场钩子、论证段、段落过渡和情绪转折。只用 30 秒样本训练分镜，30 秒之后的视觉语法很容易走样。

### 缺口二已解决：v4 Prompt 补齐“原片-后期”拆解逻辑

v4 不只是补几个标签，而是把拉片目标从“记录画面元素”改为“拆解创作者的剪辑与合成逻辑”。它会明确告诉下游：哪些是底层录屏/实拍，哪些是后期 UI、运镜、遮罩、弹窗和转场。

仍待解决的是：多段拉片结果合并后的风格、Layer 命名和重叠时间段一致性。

## 下一步优先事项

第一优先级仍是按 [端到端管线操作手册](./2026-06-17_15-50_端到端管线操作手册.md) 跑通一次 30 秒视频，用现有 C/D 两份高质量拉片样本验证从分镜到成片的可行性。

之后再补充上游样本：先用 v4 Prompt 拆解视频 E（豆包是真人炼的），再返工视频 A/B。等 v4 样本足够多，再回头判断是否需要把 C/D 也用 v4 重新拉片。

## 已知风险

最主要的风险还是偏差逐层累积：拉片阶段模型可能误判复杂图层运动；Few-Shot 阶段模型可能分不清“优秀视觉语法”和“拉片噪音”；渲染阶段再把偏差翻译成代码，最终画面就会越来越偏。

应对方式是继续坚持三件事：拉片时强制原片/后期解耦；分镜时只写标准物理字典；渲染前用素材清单和工程标准做人工检查。
