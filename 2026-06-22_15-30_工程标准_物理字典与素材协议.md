## 防偏差工程标准：闭环物理动作字典

为了防止大模型在分析视频或输出动效时凭空捏造不存在的第三方库私有 API，所有“运动与渲染”描述都必须先翻译为标准网页底层物理字典。

### 1. 空间定位

所有位置坐标一律以左上角为基准点 `[0%, 0%]`，右下角为 `[100%, 100%]`。

严禁直接使用设备相关的固定像素点来描述位置，例如 `1920x1080` 下的某个具体坐标。可以写尺寸和视觉规格，但运动位置要优先写相对百分比。

### 2. 标准物理变换字典

只允许把普通图层运动描述到这些底层物理量：

| 物理量 | 用法 |
| :--- | :--- |
| `Scale (缩放)` | 基准值为 `100%` |
| `Opacity (不透明度)` | `0%` 完全透明，`100%` 完全可见 |
| `TranslateX / TranslateY (相对平移)` | 用相对窗口宽高的百分比表示 |
| `Rotate (旋转)` | 用顺时针角度表示 |
| `Blur (高斯模糊)` | 用像素半径表示 |
| `ClipPath (裁剪遮罩)` | 采用 CSS 矩形语法，例如 `inset(0% 100% 0% 0%)` |
| `Pop-up (弹性弹出)` | 本质是 `Scale` / `Translate` 加 `Spring` 缓动，不是库名 |

### 3. 标准插值与缓动控制

| 类型 | 要求 |
| :--- | :--- |
| `Linear` | 指定秒数内匀速变化 |
| `Spring` | 必须提供 `Damping` 和 `Mass`，常用 `Damping 10-20`、`Mass 0.5-1.0` |
| `Bezier` | 必须指定控制点参数，例如 `cubic-bezier(0.25, 0.1, 0.25, 1.0)` |

### 4. 流媒体语法

`[SCREEN_RECORD]` 和 `[FOOTAGE]` 是连续视频流。它们内部的时间流逝严禁用 `Translate` 误写，必须使用流媒体语法。

| 语法 | 用途 |
| :--- | :--- |
| `Playback (内部播放)` | 描述画面里正在发生的客观动作，例如人物走动、镜头推进、车辆移动 |
| `Scroll (视口滚动)` | 专用于录屏素材，描述界面视口上下滑动或浏览列表 |
| `Cursor Move (光标滑动)` | 描述鼠标/手势指针移动 |
| `Typewriting (拟真打字)` | 描述输入框逐字输入、拼音候选、光标闪烁 |
| `Click (点击反馈)` | 描述按钮点击、页面响应、涟漪或选中反馈 |

视频流外层容器自身的出入场仍可使用标准物理变换，例如一个浏览器窗口整体 `TranslateY` 滑入。必须写清楚是“容器本身在动”，还是“容器内部的录屏在滚动/播放”。

### 5. 静态 B-roll 运镜

当 `[STOCK]` 或 `[BROWSER]` 静态图片作为全屏或近全屏 B-roll 使用时，默认使用：

`Ken Burns Effect (缓慢推拉摇移)`

也就是缓慢 `Push-in`、`Pull-out` 或 `Slow Pan`，用来打破静态图片的死板。只有确实出现多层素材差速位移时，才额外说明 `Parallax`。

### 6. 场景转场术语

场景更迭必须使用专业剪辑术语定性，不能只写“图层消失/出现”。

当前 v4 拉片 Prompt 强制使用这些转场类型：

| 术语 | 含义 |
| :--- | :--- |
| `Hard Cut (硬切)` | 瞬间替换画面 |
| `Cross Dissolve (叠化)` | 两个画面透明度交叉渐变 |
| `Push (推像)` | 新画面把旧画面顶出视窗 |
| `Masking (遮罩)` | 利用遮罩区域或运动物体边缘引出新画面 |

如果历史拉片里出现 `Fade to/from Black`、`Glitch`、`Match Cut` 等术语，可以作为补充理解，但新增 v4 拉片优先按上表归类。

## 拉片素材标签

v4 分析 Prompt 的核心标签是这五类：

| 标签 | 含义 |
| :--- | :--- |
| `[SCREEN_RECORD]` | 连续的真实屏幕操作录屏流，通常是底图 |
| `[FOOTAGE]` | 连续实拍视频片段或 CG 动画流，通常是底图 |
| `[CODE]` | 后期软件生成的 UI 元素、高亮框、文字卡片、弹窗、遮罩等 |
| `[STOCK]` | 后期插入的静态图片、插画、B-roll 空镜 |
| `[BROWSER]` | 静态网页或 App 截图 |

分镜生成和素材清单阶段仍可继续使用 `[AI_GEN]`、`[HUMAN]`，因为那两个标签描述的是“素材怎么获得”，不是拉片时画面物理本质。

## 资产追踪工程标准：统一素材 To-Do 协议

脚本中引用的所有非代码生成资产，都必须进入素材 To-Do 协议，方便 AI 检查和人类核对。

### 1. 资产分类与命名规范

每个素材编号为 `ASSET_[镜头序号]_[图层序号]`，统一存入 `视频项目/.../04_素材清单/assets/`。

| 素材分类 | 生产方式 | 状态标记 |
| :--- | :--- | :--- |
| `GENERATE_BY_CODE` | 纯代码绘制，无外部依赖 | `[AUTO]` |
| `AI_BROWSER_CAPTURE` | AI 通过浏览器自动化访问网页并截图 | `[TODO: BROWSER]` |
| `STOCK_LIBRARY` | 图库或搜索引擎检索的公共素材 | `[TODO: STOCK]` |
| `AI_GENERATED` | 提示词驱动 AI 生成的图片 | `[TODO: AI_GEN]` |
| `HUMAN_CAPTURE` | 只有人类才能获取的素材 | `[TODO: HUMAN]` |

### 2. AI 自动化执行规则

1. AI 获取或生成资产后，必须重命名为对应资产 ID 并存入 `视频项目/.../04_素材清单/assets/`。
2. 图片就绪后，将对应文档中的 `[TODO: ...]` 修改为 `[DONE]`。
3. 渲染时如发现 `[TODO]` 状态的资产文件不存在，脚本必须阻断并告警。
4. AI 原始输出必须原样保存在对应归档目录中，人工校正在独立文件中进行，防止源头污染。

### 3. 素材台账

最终选定的素材在 `视频项目/.../04_素材清单/assets/asset_ledger.json` 中记录来源和授权信息。

字段包括：`asset_id`、`source_url`、`provider`、`license`、`author`、`fetched_at`、`status`。

### 4. 推荐图库工具

| 工具 | 优先级 | 适合用途 |
| :--- | :--- | :--- |
| Pexels API | 第一优先 | 实拍照片、人物、办公、科技感 B-roll |
| Unsplash API | 第二优先 | 高质量摄影图、氛围图 |
| Pixabay API | 补充 | 插画、图标、透明感素材 |
| Openverse API / Wikimedia Commons | 补充 | 公共领域、CC 授权、历史/科学图 |
| Brave Image Search API | 补充 | 全网灵感，授权需人工复核 |

不推荐：Google AI Mode（无公开 API）、Bing Image Search API（已退役）、Google Programmable Search（已停止新客户）。
