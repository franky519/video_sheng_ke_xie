## 项目愿景与全链路生产流程

### 终极目标

给定一篇科普文字稿，通过 AI 全链路自动化，生成一部完整的、接近差评君/冲浪Plus 质量水准的科普视频。

### 人类与 AI 的分工边界

**人类只负责两件事**：
1. 提供科普选题的文字稿（内容方向、核心论点、大概讲什么）
2. 最终成片的质量审核与微调

**其余所有环节由 AI 自动完成**。

### 全链路流程

#### 第一步：分镜头脚本生成（详细说明）

分镜头脚本是整条链路的核心产物。**不是**粗略的事件流（"这里有个卡片弹出来"），而是一份用物理字典精确描述画面的完整生产文档。

**注意：分镜脚本只使用物理字典描述动画（Scale/Opacity/Blur/Spring 等），不指定具体实现库。** 库的选择属于第四步渲染阶段的决策。

**输入**：

| 输入 | 来源 | 说明 |
| :--- | :--- | :--- |
| 文字稿 | 人类提供 | 科普选题的文案内容 |
| 标杆视频 Few-Shot 拆解样本 | `02_别人视频的拆解分析/` | 差评君/冲浪Plus 等标杆视频经 Gemini 多模态拉片后的结构化脚本，AI 从中学习视觉语法 |

**拆解工具链**（用于生成 Few-Shot 样本）：
- 视频分析脚本（OpenRouter）：`03_视频抽帧与裁剪工具/2026-06-02_09-45_openrouter_analyze_30s.py`（裁剪→缩放→base64→OpenRouter API）
- 视频分析脚本（直连 Files API）：`03_视频抽帧与裁剪工具/2026-06-02_17-45_gemini_native_analyze_30s.py`（裁剪→缩放→上传→Files API）
- **当前使用的分析 Prompt（v2）**：`04_教AI拉片的提示词/2026-06-09_17-30_Gemini物理拉片分析prompt_v2.txt`
  - v2 相比 v1 的改动：删除"程序化制法与代码库参考"列；新增设计意图（用 → 连接运动与叙事目的）；素材打标分类（`[BROWSER]`/`[CODE]`/`[STOCK]`/`[AI_GEN]`/`[HUMAN]`）；静态层单独声明不在时间轴重复
- 旧版 Prompt（v1，已废弃）：`04_教AI拉片的提示词/2026-06-02_11-45_Gemini物理拉片分析prompt.txt`
- 视觉语法迁移示例：`04_教AI拉片的提示词/2026-06-04_11-40_豆包前30秒Few-Shot视觉语法迁移示例.md`

**拆解生产方案**（详见 `02_别人视频的拆解分析/2026-06-10_19-00_视频拉片分段策略实验计划.md`）：
- 渠道：OpenRouter + gemini-3.5-flash (reasoning=high)
- 分段策略：30s/段，5s 重叠缓冲，每个视频约 12 段
- 合并：OpenCode + DeepSeek V4 Pro 非交互模式合并分段结果
- 仅接受 gemini-3.5-flash 的拆解结果，gemini-2.5-flash 质量不达标已归档

**分镜脚本需要精确到**：
- 每个图层的内容、构图意图与视觉层级（观众先看哪里）
- 物理动画参数（如"Scale 0→100%, spring damping 15, mass 0.6"）
- 文字/配音与画面的时间对齐
- 情绪意图（为什么用这种节奏、这段是制造紧张感还是给观众喘息）

#### 第二步：素材清单提取（详细说明）

分镜脚本生成后，单独进行一轮解析，从中提取每个图层引用的素材，逐个归类、编号、写获取指令。这是对分镜的**后处理**，不是同步输出——让每一步的任务更聚焦，减少 AI 出错。

#### 第四步：程序化渲染合成（详细说明）

渲染阶段才引入库选型决策。输入除了分镜脚本和素材外，还需要 `06_库对比测试/conclusion.md` 中的"特效→最佳库"映射表（格式：全局汇总表 + 按库聚合统计 + 各 case Gemini 评判原文，case 1 已完成评判），确保每种特效使用实测验证过的最优库，而非 AI 凭训练数据猜测。

### 当前进度与核心缺口

**缺口一：Few-Shot 拆解样本严重不足**

| 标杆视频 | 已有高精度拆解 | 质量判断 | 缺口 |
| :--- | :--- | :--- | :--- |
| 抖音AI涨价 | 前 30 秒（OpenRouter, gemini-3.5-flash, v2 prompt） | **格式达标，质量最高**，含设计意图和素材分类标签 | 需按 30s×12 段方案拆解完整 5min |
| 差评君 | 前 30 秒（旧版 v1 prompt） | 旧版格式，缺设计意图列，已归档 | 需用 v2 prompt + 3.5-flash 重新拆解 |
| 差评君 1 分钟版 | 有，但为旧版"制作建议"格式 | **不符合物理字典标准，不可用** | 需按新标准重新拆解 |
| 冲浪普拉斯 | 几乎为零（5 分钟版仅连通性测试） | **完全不可用** | 需从头按标准拆解完整视频 |

每个标杆视频需要完整拆解（参考视频本身约 5 分钟），才能覆盖完整的视觉语法（开场钩子 + 论证段 + 段落过渡 + 情绪曲线转折）。30 秒只有开头，AI 生成的分镜在 30 秒后必然走样。

**缺口二已部分解决：v2 Prompt 补齐了设计意图和素材分类**

v2 Prompt 改进后，拆解输出已从"事件流"升级为包含设计意图的分镜：
- 每个运动描述附带 `→` 叙事目的（注意力引导、情绪效果、信息节奏）
- 素材带分类标签（`[BROWSER]`/`[CODE]`/`[STOCK]` 等），不同类型用不同描述精度
- 静态层抽离为"持久画面架构"，不在时间轴重复

仍待解决：库选型（属于第四步）、多段合并后的一致性。

**下一步优先事项**：
1. 按 30s×12 段方案完成抖音AI涨价的完整拆解（OpenRouter + 3.5-flash + v2 prompt）
2. 用 OpenCode + DeepSeek V4 Pro 合并 12 段为完整文档
3. 同步对差评君、冲浪普拉斯执行同样流程
4. 库对比测试（`06_库对比测试/`）完成剩余 case 的评判（10 个 case 中已完成 1 个）
5. 端到端验证：用完整样本作为 Few-Shot 输入，测试 AI 能否生成质量稳定的全长分镜

**已知风险：偏差逐层累积**

即使上述缺口全部补齐，仍需警惕三层偏差的传导：拆解阶段 Gemini 对复杂图层运动的理解可能有偏差（如误判多图层复合运动）→ Few-Shot 阶段 AI 可能无法区分"优秀细节"与"拉片噪音"，导致逻辑放大 → 渲染阶段偏差累积使最终画面失控。应对方式见素材协议中的"草稿隔离"规则。

## 防偏差工程标准：闭环物理动作字典

为了防止大模型在分析视频或输出动效时凭空捏造（Hallucinate）不存在的、或者不兼容的第三方库（如 Moviepy/Remotion）私有 API 导致其他 AI 编写代码时报错，我们规定所有“运动与渲染”的描述必须翻译为**“标准网页底层物理字典”**。

在提示词与脚本中，只允许大模型映射至以下通用属性：

### 1. 空间定位（基准百分比）
* 所有的位置坐标一律以左上角为基准点 `[0%, 0%]`，右下角为 `[100%, 100%]`。
* 严禁直接使用设备相关的固定像素值（如 `1920x1080` 下的特定像素点），必须写成相对百分比坐标，例如 `位置：[50%, 50%]`（居中）。

### 2. 标准物理变换字典（CSS/JS Primitives）
动效拉片中对“运动”的程序化描述，必须直接映射到如下标准物理量：
* `Scale (缩放)`：以基准值 `100%` 进行变化。
* `Opacity (不透明度)`：取值范围 `0%`（完全透明）至 `100%`（完全可见）。
* `TranslateX / TranslateY (相对平移)`：用相对窗口宽高的百分比表示（例如：`TranslateX 从 100% 移至 0%` 表示自右滑入）。
* `Rotate (角度旋转)`：用顺时针角度度数表示（例如 `rotate(-15deg)`）。
* `Blur (高斯模糊)`：用模糊像素半径表示（例如 `blur(15px)`）。
* `ClipPath (裁剪遮罩)`：采用 CSS 的矩形滑块语法（例如 `clipPath: inset(0% 100% 0% 0%)` 表示水平方向向右擦除露出）。

### 3. 标准插值与缓动控制（Easing & Springs）
* **线性插值（Linear）**：即在指定秒数内匀速变化。
* **弹性物理函数（Spring）**：必须提供两个关键控制量：
  * `Damping (阻尼系数)`：一般介于 `10 - 20` 之间，越低越弹，通常 `12` 为经典回弹，小于 `10` 容易产生反复震荡。
  * `Mass (质量系数)`：一般为 `0.5 - 1.0`，越低越轻，弹出速度越快。
* **贝塞尔曲线控制（Bezier）**：必须指定起止控制点参数（例如 `cubic-bezier(0.25, 0.1, 0.25, 1.0)`）。

## 资产追踪工程标准：统一素材 To-Do 协议

为了把抽象的“镜头描述”彻底转为工程上可被 AI 检查和人类核对的待办清单，脚本中引用的所有非代码生成资产，都必须遵循以下**资产 To-Do 协议**。

### 1. 资产分类与命名规范
每个素材编号为 `ASSET_[镜头序号]_[图层序号]`（如 `ASSET_01_02.png`），统一存入 `01_文字稿与原始素材/assets/`。

| 素材分类 | 生产方式 | 状态标记 |
| :--- | :--- | :--- |
| **`GENERATE_BY_CODE`** | 纯代码绘制（CSS/SVG/渐变背景），无外部依赖 | `[AUTO]` |
| **`AI_BROWSER_CAPTURE`** | AI 通过浏览器自动化访问实际网页并截取（如 ChatGPT 界面、新闻页面） | `[TODO: BROWSER]` |
| **`STOCK_LIBRARY`** | 搜索引擎/图库检索的公共素材（实拍 B-roll、图表等） | `[TODO: STOCK]` |
| **`AI_GENERATED`** | 提示词驱动 AI 生成的图片（如 Midjourney），需记录 Prompt | `[TODO: AI_GEN]` |
| **`HUMAN_CAPTURE`** | 确实只有人类才能获取的素材（极少数情况） | `[TODO: HUMAN]` |

### 2. AI 自动化执行规则

1. **自动归网与命名**：AI 获取或生成资产后，必须重命名为对应资产 ID 并存入 `assets/` 目录。
2. **状态自动勾选**：图片就绪后，将对应文档中的 `[TODO: ...]` 修改为 `[DONE]`。
3. **未就绪告警**：渲染时如发现 `[TODO]` 状态的资产文件不存在，脚本必须阻断并告警。
4. **草稿隔离**：AI 原始输出必须原样保存在 `AI_原始输出/` 目录，人工校正在独立文件中进行，防止源头污染。

## 增量问答 / QA

### 2026-06-09 20:38：Google AI Mode 能否作为素材自动找图 API？

结论：暂时不能把 Google Search 前台的 **AI Mode** 当成可直接调用的官方找图 API。官方资料把 AI Mode 描述为 Google Search 里的用户搜索体验，它可以在前台给出视觉结果、支持追问和视觉探索，但未看到面向开发者的独立 AI Mode API。

工程上可选的 Google 路径有三类：

| 路径 | 是否适合本项目素材清单 | 说明 |
| :--- | :--- | :--- |
| Gemini API / Vertex AI 的 Google Search grounding | 适合辅助判断，不适合直接批量下图 | 它能让模型联网检索、返回引用来源，但核心产物是带依据的文本回答，不是稳定的图片候选列表与下载链接。 |
| Programmable Search / Custom Search JSON API 图片搜索 | 老项目短期可用，新项目不建议依赖 | 该 API 支持 `searchType=image` 返回图片搜索 JSON，但 Google 官方文档显示它已不向新客户开放，已有客户也需要在 2027-01-01 前迁移。 |
| Agent Search（原 Vertex AI Search）网站图片搜索 | 适合“限定来源”的素材库检索 | 可以对指定网站数据做图片搜索，文本查询可返回图片结果；但它不是全网 Google 图片搜索，也不是 AI Mode。适合把 Wikimedia、Pexels、公司官网等许可明确的网站纳入候选来源后检索。 |

因此，`[TODO: STOCK]` 的自动化方案不应写成“调用 Google AI Mode”。更稳的设计是：先由 Gemini 把素材描述改写成搜索关键词和筛选条件，再调用可审计的素材来源 API 或 Agent Search 限定站点检索，最后保存图片 URL、来源页、授权信息、下载时间和人工复核状态。AI Mode 可以继续作为人工快速找灵感的前台工具，但不应作为自动化生产链路的核心接口。

### 2026-06-09 22:03：怎样复刻“Google AI Mode 一次给 6-10 张图让我挑”的体验？

目标不是让 AI 直接替人选定最终素材，而是让它先为每个 `[TODO: STOCK]` 生成一个“图片候选池”：一次返回 6-10 张候选缩略图，每张图带来源页、可下载图片地址、作者/平台、授权信息、尺寸、匹配理由和风险提示。人只需要看候选卡片并选择一张，AI 再自动下载、重命名为 `ASSET_xx_xx.png`，并把清单状态改成 `[DONE]`。

推荐工具组合如下：

| 工具 | 适合用途 | 工程判断 |
| :--- | :--- | :--- |
| Pexels API | 实拍照片、人物、办公、科技感 B-roll | 最适合作为第一优先级。官方 API 支持图片和视频搜索，结果质量稳定，适合快速给出多张候选。需要按 Pexels 规则保留 attribution。 |
| Unsplash API | 高质量摄影图、氛围图、科技/办公/自然类素材 | 候选质量高，适合做第二来源。官方 API 支持关键词搜索、动态裁切 URL 和下载追踪，但要求使用返回的图片 URL、触发 download endpoint、保留作者和 Unsplash 归因。 |
| Pixabay API | 插画、图标、透明感素材、照片补充 | 适合作为补充来源。官方 API 返回 `previewURL`、`webformatURL`、`largeImageURL`、`pageURL` 等字段，方便做候选卡片和下载。 |
| Openverse API / Wikimedia Commons | 版权更可审计的公共领域、CC 授权、历史/科学/机构图 | 适合“科普视频里需要真实机构、历史图片、公共领域图片”的场景。缺点是视觉质量参差不齐，需要 AI 预筛 + 人工确认授权。 |
| Brave Image Search API | 接近搜索引擎的全网图片候选 | 适合找“全网灵感”和补足图库找不到的具体物件。官方图片搜索接口可返回大量图片结果，但授权风险更高，不能自动直接入库，必须进入人工复核。 |
| SerpApi Google Images API | 最像 Google 图片搜索的第三方接口 | 体验接近 Google Images，可作为实验项。但它不是 Google 官方 API，合规和长期稳定性要单独评估，不建议作为默认生产依赖。 |
| Tavily Search `include_images` | 文本检索时顺带拿图 | 适合“先找资料页，再顺手提取页面相关图片”的场景，不适合作为纯图片素材库主入口。 |

| 工具 | 原因 |
| :--- | :--- |

| Google Programmable Search / Custom Search JSON API 图片搜索 | 支持 `searchType=image`，但官方已不向新客户开放，老客户也需要迁移，适合了解历史方案，不适合作为新工程核心。 |
