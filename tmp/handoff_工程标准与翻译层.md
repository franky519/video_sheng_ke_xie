### 待处理

**P002: 素材标签到资产协议翻译层设计**（`_proposals/P002_素材标签到资产协议翻译层设计.md`，状态 GRILLED）

这是下一个 session 最需要推进的问题。核心矛盾：

当前工程标准里素材标签（五类：SCREEN_RECORD / FOOTAGE / CODE / STOCK / BROWSER）描述的是"画面里是什么类型的视觉元素"。资产协议（五类获取方式）描述的是"这东西怎么获得"。看起来一一对应，实际有三个断层：

1. **[FOOTAGE] → 资产协议无对应获取路径。** AI 集体涨价拆解文档里有大量 FOOTAGE 标注——健身房实拍、CG 立方体旋转动画、ofo 单车山实拍、数据中心航拍。但资产协议里没有"生成一段 CG 动画"的获取方式，AI_GENERATED 只覆盖静态图片。执行 Agent 拿到 FOOTAGE 后不知道这段素材是该去 YouTube 搜还是用 SeeDance 生成。

2. **[CODE] 压扁了实现复杂度。** 高亮框（inset 展开）和网络图概念演示动画（3×3 网格节点互连、人脸转 AI 头像）都被标成 CODE——但前者是 Remotion 里的 CSS clip-path，后者可能需要 Manim（3Blue1Brown 的库）才能做好。标签把轻量 UI 和重度概念动画混在了一起。

3. **[BROWSER] 的"构造场景"需求被忽略。** 拆解文档里"已用完 Free 额度，系统将使用轻量模型"的灰字报错——分析已有视频只需要描述它。但如果要新做一个类似画面，你得先决定报错文案、构造对话上下文、触发那个报错状态，然后才能截图。"先构造再截图"这多出来的一步当前标签体系完全没覆盖。

根本原因：素材标签用的是**分析视角**（看已有视频、描述画面里有啥），资产协议用的是**生产视角**（从零做视频、把每层描述变成可执行任务）。中间缺了翻译层——从分析语言到生产策略的转译规则。

P002 选定的方案是两阶段：
- 第一阶段：在工程标准里加一个"翻译路径"章节（不改标签定义本身，先补路径映射）
- 第二阶段：如果翻译路径暴露出标签需要拆分，再单独提 proposal 改标签和 v4 Prompt
- 但注意：P002 里还有一条任务"在依赖关系图中补标格式继承"，这条已被 P003 覆盖完成

## 遇到的核心认知问题

本 session 前期我在理解本项目时犯了一个关键错误——把 v4 Prompt 当成分镜脚本格式的上游定义点。实际逻辑是反过来的：

> v4 Prompt 只是**生成拆解文档的工具**。拆解文档的 `时间轴 | 底层原片事件 | 后期干预手段 | 音频` 四列表格才是真正定型后的产物格式，是所有新增分镜脚本必须模仿的格式模板。

这个关系之前没有在任何文档里显式写出来（只在依赖关系图链 1 里隐式存在），导致每个新 Agent 都会犯同样的误解。现在已在项目总览、目录导航、依赖关系图三个位置补上了。
