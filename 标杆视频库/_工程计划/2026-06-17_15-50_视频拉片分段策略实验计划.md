## 背景

早期 v2 拉片 Prompt 已验证 30 秒分段方案可行；当前新增或返工拉片统一改用 v4 原片/后期解耦 Prompt。完整视频实际 **223 秒（约 3 分 43 秒）**，模型单次输出上限 ~65536 tokens，按 30s 约 8000 tokens 的输出量估算，全量一次输出不可能。必须分段。

### 渠道决策

**确定使用：OpenRouter + gemini-3.5-flash (reasoning=high) + 裁剪短片 base64 上传**

原因：
- Google 直连 Files API 技术上更优（原生视频处理、支持长视频全量上传），但当前**未配置 V 卡付费**，免费额度不可靠（3.5-flash 频繁 503，pro 模型 429）
- OpenRouter 走 base64 编码，30 秒裁剪片段已验证可行（~6.4 MB payload，响应快，¥0.60/次，质量最高）
- 5 分钟完整视频 base64 不可行（~64 MB payload 超时），因此必须裁剪为短片逐段上传

> **搁置：路径 B（全量上传 + 分段 Prompt via Files API）**
>
> 依赖 Google 官方付费 API。待配置 V 卡后可重新评估。

## 确定方案：30s 分段 + 5s 重叠缓冲

### 为什么 30s

- 已验证 30s 输出质量最高（完整、不截断、设计意图有叙事深度）
- 费用分析表明总成本与段长无关（费用 97% 来自输出 token，5min 总输出量恒定 ~84000 tokens，不论怎么切）
- 30s 单段 base64 体积 ~6.4 MB，OpenRouter 处理无压力

### 为什么加重叠缓冲

30s 硬切会导致段边界附近的内容可能被遗漏或截断（某个动效跨越了 0:30 的时间点）。加 5s 的重叠缓冲确保边界内容在前后两段都被分析到。

### 分段方案

每段 30s，步进 25s（即 5s 重叠）：

| 段号 | 时段 | 步进说明 |
|:---|:---|:---|
| S01 | 0:00 - 0:30 | 起始段（已有结果，作为基线） |
| S02 | 0:25 - 0:55 | 与 S01 重叠 0:25-0:30 |
| S03 | 0:50 - 1:20 | 与 S02 重叠 0:50-0:55 |
| S04 | 1:15 - 1:45 | |
| S05 | 1:40 - 2:10 | |
| S06 | 2:05 - 2:35 | |
| S07 | 2:30 - 3:00 | |
| S08 | 2:55 - 3:25 | |
| S09 | 3:20 - 3:43 | 末尾段（23s） |

共 **9 段**，预估总费用 ~¥5.40（9 × ¥0.60）。S01 已有基线结果。

### 两阶段流水线

```
阶段 1：分段拉片（OpenRouter + gemini-3.5-flash）
  ├── S01: 0:00-0:30  → S01_0m00s-0m30s.md （已有基线）
  ├── S02: 0:25-0:55  → S02_0m25s-0m55s.md
  ├── ...
  └── S09: 3:20-3:43  → S09_3m20s-3m43s.md

阶段 2：合并（OpenCode + DeepSeek V4 Pro，非交互模式）
  ├── 输入：9 个分段拉片结果
  ├── 处理：按重叠区域对齐，去重，统一 Layer 编号，拼接为完整文档
  └── 输出：完整拉片分析_merged.md
```

## 脚本：已创建

### 阶段 1 脚本：`03_视频抽帧与裁剪工具/2026-06-10_21-00_openrouter_segment_analyze.py`

支持单段和批量两种模式：

```bash
# 单段模式
python3 2026-06-10_21-00_openrouter_segment_analyze.py \
  --video ../02_别人视频的拆解分析/抖音AI涨价/raw_video/01_...mp4 \
  --output-dir ../02_别人视频的拆解分析/抖音AI涨价/分段拉片结果/ \
  --start 25 --end 55

# 批量模式（自动检测视频时长，生成所有分段，跳过已有结果）
python3 2026-06-10_21-00_openrouter_segment_analyze.py \
  --video ../02_别人视频的拆解分析/抖音AI涨价/raw_video/01_...mp4 \
  --output-dir ../02_别人视频的拆解分析/抖音AI涨价/分段拉片结果/ \
  --batch
```

特性：已有结果自动跳过（S01 基线不会重复执行），段间 3s 冷却避免限频。

### 阶段 2：合并脚本

使用 OpenCode 非交互模式调用 DeepSeek V4 Pro：
```bash
opencode -p "请将以下 9 个分段拉片结果合并为一份完整文档。\
重叠区域（每段首尾 5s）以描述更详细的一方为准。\
统一 Layer 编号，保持持久画面架构只出现一次。\
输出为单个 Markdown 文件。" \
  S01_*.md S02_*.md ... S09_*.md
```

（具体命令格式需要根据 OpenCode 的非交互模式参数调整）

## 已有实验结果汇总

| 时间 | 模型 | 渠道 | 视频段 | 段长 | 输入 tokens | 输出 tokens | 费用 | 完整度 | 备注 |
|:---|:---|:---|:---|:---|:---|:---|:---|:---|:---|
| 06-09 18:58 | gemini-2.5-flash | 直连 | 0:00-0:30 | 30s | - | - | 免费 | 截断~12.5s | v2 prompt 首次测试 |
| 06-09 19:05 | gemini-2.5-flash | 直连 | 0:00-0:30 | 30s | - | - | 免费 | 完整 30s | 第二次运行 |
| 06-09 19:08 | gemini-3.5-flash | OpenRouter | 0:00-0:30 | 30s | 4582 | 8390 | ¥0.60 | 完整 30s | **S01 基线**，质量最高 |
| 06-09 19:20 | gemini-3.1-pro | OpenRouter | 0:00-0:30 | 30s | - | - | - | 失败504 | 推理超时 |
| 06-10 19:30 | gemini-3.5-flash | OpenRouter | 全量 5min | - | - | - | - | 失败超时 | base64 payload 过大 |

### 已验证结论

1. **gemini-3.5-flash (OpenRouter, reasoning=high)** 是当前最适合视频拉片的模型组合
2. **30s 分段**是质量、成本、工程复杂度的最优平衡点
3. **OpenRouter base64 可行上限**：30s 裁剪片段（~6.4 MB）可行；5min 全量不可行
4. **Google 直连 Files API 技术更优**，待配置付费后可重新评估
