# P002 素材标签到资产协议：翻译层设计

状态：GRILLED

## 问题观察

讨论工程标准重构时发现了一个更深的结构性问题：**素材标签和资产协议之间缺了一层翻译。**

当前工程标准里，素材标签（五类：`[SCREEN_RECORD]`、`[FOOTAGE]`、`[CODE]`、`[STOCK]`、`[BROWSER]`）定义的是"这东西在画面里是什么类型的视觉元素"。资产协议（五类获取方式：`GENERATE_BY_CODE`、`AI_BROWSER_CAPTURE`、`STOCK_LIBRARY`、`AI_GENERATED`、`HUMAN_CAPTURE`）定义的是"这东西怎么获得"。

看起来是对应的，但实际上三个标签的执行路径有断层：

**`[FOOTAGE]` → 资产协议无对应路径。** 比如 AI集体涨价拆解文档里"透明立方体在黑色背景中旋转"标注为 `[FOOTAGE]`——它是 CG 动画，不是现有的实拍素材。资产协议里没有"生成一段 CG 动画"的获取方式，`AI_GENERATED` 只覆盖静态图片。执行 Agent 拿到 `[FOOTAGE]` 后，不知道这段动画是去 YouTube 搜还是用 SeeDance 生成。

**`[CODE]` 压扁了实现路径。** 高亮框（`inset` 展开）和数据驱动图表（柱状图配 60 亿/150 亿数据）都被标成 `[CODE]`，但前者纯 Remotion 画，后者需要先找到数据来源再画。比图表更重的是概念演示动画——比如拆解文档里"3×3 网格节点互连、人头转 AI 头像、绿线消失"那样的示意动画——它跟高亮框共享 `[CODE]` 标签，但工作量差几个数量级、需要的库可能完全不同（Remotion vs Manim）。

**`[BROWSER]` 的"构造场景"需求。** 拆解文档里"已用完 Free 额度，系统将使用轻量模型"的灰字报错——在分析已有视频时只需要描述即可。但如果要新做一个类似画面，得先决定报错文案、构造对话上下文、触发那个报错状态，然后才能截图。纯截图变成了"先构造场景再截图"——这多出来的步骤当前标签体系完全没覆盖。

根本原因：素材标签设计时用的是**分析视角**（看一个已有视频，描述画面里有啥），而资产协议设计时用的是**生产视角**（自己从零做一个视频，把每一个画面描述变成可执行的任务）。两者之间缺的翻译层就是"从分析视角到生产视角的转译规则"。

此外，依赖关系图和目录导航目前没有显式标注"拆解文档的表格格式 → 分镜脚本的格式模板"这个继承关系，导致后续 Agent 容易把拆解文档仅当成 Few-Shot 样本而非格式源头。

## 变更意图

设计素材标签到资产协议的翻译层，使分镜脚本的描述能从"事后分析语句"升级为"带执行策略的生产指令"。具体：

1. 审视素材标签是否需要拆分或改名（比如 `[CODE]` 是否拆成"简单 UI 元素"和"数据/概念动画"两类）
2. 审视资产协议是否需要扩充（比如加 `AI_VIDEO_GENERATE` 覆盖 `[FOOTAGE]` 的 CG 动画生成路径，加 `SCENE_CONSTRUCT` 覆盖 `[BROWSER]` 的"先构造再截图"）
3. 为每个素材标签定义至少一种到资产协议的翻译路径（"如果是 A 类型，可以走 B 方式获取；如果是 C 类型，走 D"）
4. 在依赖关系图中补标"拆解文档→分镜脚本"的格式继承关系，在目录导航中补一句说明

## 选定方案

两阶段方案：

**第一阶段（本次）：** 在工程标准中加一个"翻译路径"章节，用表格形式列出每个素材标签 → 资产协议的默认映射 + 变体场景。不改标签定义本身，先补路径。

**第二阶段（后续）：** 如果翻译路径暴露出标签体系需要拆分（比如 `[CODE]` 确实拆分、`[FOOTAGE]` 需要细分 CG vs 实拍），再单独提 proposal 改标签定义和 v4 Prompt。

第一阶段先落地验证——路径映射本身可能已经解决了 80% 的问题。
