# P004 素材标签三层粒度框架与转译层设计

状态：GRILLED

> 2026-06-24 brainstorming 产出。验证进度持续更新中。

---

## 一、问题起源

P002 发现素材标签（拆解视角）和资产协议（生产视角）之间存在断层。深入讨论后发现本质是三层粒度分离不够：

- 拆解时看已有视频，描述的是"画面里有什么"
- 执行时从零做视频，需要的是"怎么获取/制作这东西"
- 中间缺了一层翻译——把分析语言转成生产策略

AGENTS.md 原有的固定 proposal 模板已在本次讨论中废除，本文件由 brainstorming 自然产出。

---

## 二、三层框架

```
第一层：拆解粒度（素材标签）
  看已有视频时，画面里每样东西标什么标签
  旧体系 5 类，问题：CODE 压扁复杂度、FOOTAGE 未区分 CG/实拍

第二层：翻译层
  每种标签 → AI 执行时的 Prompt/指令
  "这东西在画面里是什么" → "怎么制作/获取它"
  本质是一组执行指引 Prompt，不是静态映射表

第三层：执行粒度
  每种获取路径下，AI 真正执行时需要什么输入/输出规范
  例如 BROWSER 不只是截图——要先构造场景状态
```

## 三、素材标签重分类

### 变更内容

| 旧标签 | 改为 | 原因 |
|--------|------|------|
| `[FOOTAGE]` | `[FOOTAGE_REAL]` | 实拍空镜（航拍、工厂、街景）— 走视频素材 API |
| `[FOOTAGE]` | `[FOOTAGE_CG]` | CG 动画（3D 立方体、概念动画）— 走 AI 视频生成 |
| `[CODE]` | `[CODE_UI]` | 纯代码可绘制 UI 元素：高亮框、文字卡片、弹窗、进度条、遮罩、鼠标指针 |
| `[CODE]` | `[CODE_DATA]` | 需要数据来源或算法设计的动画：柱状图、网络拓扑、概念演示动画 |
| `[BROWSER]` | 不拆 | 标签不变，翻译层增加"构造场景"步骤 |
| `[SCREEN_RECORD]` | 不动 | 连续录屏 |
| `[STOCK]` | 不动 | 静态图片，图库 API |

### 新标签体系（7 类）

1. `[SCREEN_RECORD]` — 连续真实屏幕操作录屏
2. `[FOOTAGE_REAL]` — 实拍视频片段
3. `[FOOTAGE_CG]` — CG 动画
4. `[CODE_UI]` — 纯代码 UI 元素
5. `[CODE_DATA]` — 数据驱动/算法动画
6. `[STOCK]` — 静态图片/插画/空镜
7. `[BROWSER]` — 静态网页/App 截图（翻译层加构造场景步骤）

## 四、翻译路径映射

| 标签 | 翻译路径 |
|------|---------|
| `[SCREEN_RECORD]` | Playwright 录屏（cookie 注入） → webm → Remotion 后期叠加 |
| `[FOOTAGE_REAL]` | 视频素材 API 搜索 → 裁剪 → 入库 |
| `[FOOTAGE_CG]` | AI 视频生成（SeedDance 等）→ Prompt 设计 → 指定时长 → 生成 → 入库 |
| `[CODE_UI]` | Remotion/CSS 直接绘制，不产生外部素材文件 |
| `[CODE_DATA]` | 先获取数据来源 + 相关图片 → 再根据数据生成动画 |
| `[STOCK]` | 图库 API（Pexels/Unsplash/Pixabay）搜索 → 入库 |
| `[BROWSER]` | 构造场景（决定报错文案、触发特定状态）→ 截图 → 入库 |

## 六、验证详情

### 6.1 SCREEN_RECORD — ✅ 已验证（含组装）

#### 执行路径

SCREEN_RECORD 有两条子路径，本次两条都已验证：

**路径 A：真实录制（Playwright + cookie 注入）**
1. 用户手动登录豆包一次
2. Playwright `connect_over_cdp` 从已登录 Chrome 导出 `storageState` JSON
3. headless 模式加载 `storageState` → 访问 doubao.com → 自动为登录态
4. Playwright 录制操作 → 输出 webm

**路径 B：Remotion 复刻 UI（理论可行，未实测）**
- Remotion 直接画豆包 UI 组件 → 模拟交互
- 完全可控，不需要联网，但工作量较大

**组装层（路径 A + B 共享）：**
5. webm 加载到 Remotion 为底片层
6. 叠加 `[CODE_UI]` 后期处理（推镜、绿框、弹窗）
7. `pnpm render` → mp4

#### 验证过程

录制场景：打开 doubao.com → 在输入框逐字输入"大模型的免费额度，好像越来越不够用了？"

组装场景：复刻差评君 Sequence 1 (00:00-00:04) 做法
- Layer 0：深灰网格背景 (#1A1A1A + #333333 网格)
- Layer 1：视频容器，Slow Push-in (Scale 100%→115%)，8 秒内完成
- Layer 2：绿色扫描框 (#00FF66, 3px border, glow)，打字完成瞬间 opacity 0→1，clipPath 从左到右擦出

#### 踩坑记录

| 问题 | 原因 | 解法 |
|------|------|------|
| WSL 下 `headless=False` 窗口闪退 | WSL GUI 不稳定 | 改用 `headless=True` |
| CDP 连接 `ECONNREFUSED` | Chrome 启动缺少 `--user-data-dir` | 加上 `--user-data-dir="C:\temp\chrome_debug"` |
| `launch_persistent_context` 不支持 `storage_state` | API 不对 | 改用 `browser.new_context(storage_state=...)` |
| `page.goto` 超时（networkidle/load） | 豆包持续 AJAX 请求 | 改用 `wait_until="commit"` |
| Remotion 找不到 public 文件 | webpack bundle 临时目录未复制 static | 用 `staticFile()` 替代裸路径 |
| pnpm install 超时 | npm registry 慢 | 重试即可，通常第二次快 |

#### 产物

- 原始录屏: `视频项目/01_.../2026-06-24_SCREEN_RECORD测试_豆包模式切换.webm`
- 组装后: `视频项目/01_.../2026-06-24_SCREEN_RECORD组装验证_SlowPushIn+绿框.mp4`（4.6MB）
- Remotion 工程: `_探索/P004/组装验证/`
- 导出 cookies 脚本: `/tmp/export_doubao.py`
- 录制脚本: `/tmp/record_typing.py`

#### 局限性

- Cookies 会过期，需定期重新导出
- 页面结构变化时 CSS 选择器可能失效
- 绿框等 CODE_UI 层的精确定位需要人工调整（对着录屏判断"免费额度"在哪个像素位置）
- 当前仅验证了 8 秒片段，长视频的编排复杂度未评估

### 6.4 STOCK — ⏸️ 需 API Key

三个免费图库 API 已调研：

| API | 免费额度 | 图片数 | 视频 |
|-----|---------|--------|------|
| Pexels | 200次/h, 2万/月 | 数百万 | ✅ |
| Pixabay | 100次/min | 570万+ | ✅ |
| Unsplash | 50次/h (demo) | 850万+ | ❌ |

推荐 Pexels：额度最慷慨，图片+视频都有，注册即得 Key。与 STOCK、FOOTAGE_REAL 可共用同一 API。

### 6.5 FOOTAGE_REAL — ⏸️ 需 API Key

Pexels 同时支持视频搜索，与 STOCK 一起测。拿到 Key 后写一个脚本搜"server room"、"data center"等关键词验证。

### 6.6 FOOTAGE_CG — ⏳ 理论可行

SeedDance 等 AI 视频生成模型能力足够。需要后续设计 Prompt 模板（风格、时长、分辨率要求），尚未实测。

### 6.7 CODE_DATA — ⚠️ 技术可行，视觉不可用（搁置）

三次测试（Remotion 手写、Remotion AI 设计、data2motion 集成）均跑通了从数据到 MP4 的技术管道。核心结论：

**能做到**：AI 写 spec/代码 → 渲染图表动画 MP4（三种方式都验证了）

**做不到**：
- 视觉效果远不到差评君水准，无法直接嵌入 05_渲染工程成片
- 从旁白数据点到画面设计之间缺一层"视觉导演能力"——目前所有产出都像数据报表，不像科普视频的有机组成部分
- data2motion 图表风格和 Remotion 科技暗色风不兼容

**建议**：先搁置。等 `[SCREEN_RECORD]` + `[BROWSER]` 已经验证可行的链路跑通完整视频后，再回来升级 `[CODE_DATA]` 的视觉质量。

详细探索记录见 `_探索/P004/2026-06-24_CODE_DATA执行层探索.md`。
